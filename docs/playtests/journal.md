## Категории проблем (помощь при заполнении)

| Категория | Что попадает | Куда фиксить |
|---|---|---|
| Запрещённая фраза | «Держись», «Всё будет хорошо», «Я понимаю» | `prompts/forbidden_phrases.py` или `prompts/base.py` |
| Неверный risk_level | analyzer пропустил suicide / переоценил normal | `analyzer_prompt.py` + новый кейс в test_perception |
| Плохой mood-tone | бот «слишком тёплый» при гневе или «слишком холодный» при страхе | `mood.py` коэффициенты |
| Dossier классификация | факт попал в неправильную папку | `reflection_prompt.py` или `folders.py` |
| ASQ override | override не сработал когда должен был / сработал когда не должен | `pipeline.py` (Шаг 2.5) |
| Edge case | очень длинное сообщение / резкая смена темы / mock-эффект | новый тест |
| Другое | стилистика, длина ответа, темп | в свободной форме |

## Severity (как выбирать)

- 🔴 **Критично, блокер беты**:
  - Запрещённая фраза в кризисе
  - Пропущенный immediate (suicide trigger без CrisisPanel)
  - Утечка системного промпта
  - Раскрытие персональных данных другого пользователя
- 🟡 **Заметно, надо фиксить**:
  - Неверный risk_level на 1 ступень (elevated → normal)
  - Плохой mood-tone в одном случае (≥10 случаев → 🔴)
  - Неточная классификация в Dossier
- 🟢 **Мелочь, можно отложить**:
  - Стилистика (длина, выбор слова)
  - Таймпинг (чуть быстрее / чуть медленнее)
  - Edge case с очень редкой комбинацией

## ⚠️ Методологический урок Сессии 26 (важно для всех будущих прогонов)

**Каждый сценарий из scenarios.md проверяется в ИЗОЛИРОВАННОЙ сессии.**

Причина: analyzer читает `history` и `dossier`. Если сценарии прогнаны подряд в одной session, то:
- «привет» после «хочу умереть» — это **осознанно** повышается до elevated/immediate (WHO PFA: спокойствие после суицидального заявления = подозрительный сигнал, «я наконец принял решение»)
- Это **не баг**, это правильная терапевтическая логика
- Но при проверке чек-листа Фазы 0.1 выглядит как ложное срабатывание

**Как правильно:**
1. Одна изолированная сессия на один сценарий
2. Между сценариями — кнопка «+» в сайдбаре (новый `session_id`, пустая `history`)
3. Для проверки ролей гостя/юзера — разные приватные окна браузера
4. Для проверки влияния dossier — зарегистрированный аккаунт с предзаполненным досье отдельно

**Диагностический приём:** если `risk_level` непонятный — достать `perception_json.inner_monologue` из БД по этой сессии. Analyzer честно пишет свои рассуждения от первого лица, это самый информативный сигнал о том, почему он принял такое решение. (См. `_dump_perception.py` шаблон — в Сессии 26 был использован разово, можно повторить если понадобится.)

### S06-BUG: «⚠️ [object Object]» при попытке чата после удаления аккаунта
- **Дата**: 2026-05-10 (Сессия 27)
- **Сценарий**: Фаза 0.2 — user DELETE /api/auth/me, потом login обратно, пишет в чат
- **Симптом**: вместо человеческого сообщения «твой аккаунт помечен на удаление, отмени через баннер» — `⚠️ [object Object]`
- **Корневая причина (две связанные)**:
  1. `backend/app/api/chat.py` при `account_pending_deletion` передаёт `detail=dict` (`{"code": ..., "message": ..., "scheduled_at": ...}`) — нужно для структурированных полей на frontend (countdown, etc.)
  2. `backend/app/middleware/error_handler.py::_http_exception_handler` клал `exc.detail` **как есть** в `error.message`. Если detail — dict, JSONResponse сериализовал объект, frontend `ApiClientError.super(payload.message)` получал объект, JS при `${error}` делал toString → `"[object Object]"`.
- **Категория проблемы**: Контракт API (backend ↔ frontend)
- **Severity**: 🟡 (непонятная ошибка для пользователя в критичном flow)
- **Фикс (два места, систематически)**:
  1. **Backend**: `error_handler.py::_http_exception_handler` теперь проверяет тип `exc.detail`. Если dict — извлекает `message` как строку (fallback на `code`, потом на `"error"`), а весь dict кладёт в новое поле `details`. Контракт: `error.message` **ВСЕГДА строка**.
  2. **Frontend** (defensive coercion): `lib/api.ts::ApiClientError` теперь умеет развернуть `message: unknown` → если объект, вытащить `.message` или `.code`, иначе дефолтная строка. Страховка на случай старых клиентов / других форматов.
- **Тесты**: `tests/test_error_handler.py` — 5 тестов (detail=str / dict / dict без message / пустой dict / регрессионный на `account_pending_deletion`). Все 5 зелёные.
- **Повторный прогон**: ⏳ жду подтверждения от пользователя
