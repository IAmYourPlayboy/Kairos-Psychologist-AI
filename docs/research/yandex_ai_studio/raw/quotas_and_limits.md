## Принципы

- **Квоты** — организационные ограничения, **изменяются по запросу в техподдержку**.
- **Лимиты** — технические ограничения архитектуры Yandex Cloud, **изменению не подлежат**.

## Квоты (можно увеличить через техподдержку)

### 🚨 Генерация текста — критично для Кайроса

| Ограничение | Значение | Для нашего use case |
|---|---|---|
| **Количество одновременных генераций в синхронном режиме** | **10** | 🚨 **Критичный потолок** |
| Запросов в секунду, асинхронный режим (запрос) | 10 | резерв на ReflectionAgent |
| Запросов в секунду, асинхронный режим (получение ответа) | 50 | |
| Запросов в час, асинхронный режим (запрос) | 5 000 | |
| Запросов в секунду на токенизацию | 50 | бесплатный токенизатор |

> 🚨 **Анализ для Кайроса:**
> На каждое сообщение пользователя `/api/chat` делает **2 синхронных LLM-вызова** (analyzer + main).
> При лимите **10 одновременных** — это потолок ~5 пользователей, которые пишут одновременно. Шестой и далее встают в очередь.
> Для закрытой беты на 20–50 человек этого хватит. Но при публичном запуске — придётся писать в техподдержку с просьбой увеличить квоту.
> **Зафиксировать в чек-листе деплоя (Блок E)**: запросить увеличение квоты до 50–100 одновременных перед публичным запуском.

## Запрос увеличения квоты

> Если необходимо больше ресурсов — отправить запрос в техподдержку, указать какие квоты нужно увеличить и на сколько.

**Что мы будем просить перед публичным запуском (заметка):**
- Одновременные синхронные генерации: **10 → 50** (минимум для 25 одновременных пользователей)
- При успехе — ещё раз поднимать по мере роста аудитории
