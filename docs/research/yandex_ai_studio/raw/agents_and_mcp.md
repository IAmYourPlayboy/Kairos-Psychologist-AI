## AI-агенты

> AI-агенты — это современный подход к построению приложений на основе ИИ.
> Расширяют возможности больших языковых моделей и служат основой для построения чат-ботов, ассистентов и голосовых интерфейсов.

Состоят из **четырёх ключевых компонентов**:
1. **LLM** — сама языковая модель с зафиксированными настройками.
2. **Инструкция (prompt)** — описывает поведение и роль агента.
3. **Инструменты (tools)** — позволяют использовать внешние возможности (API, функции, поиск).
4. **Память (memory)** — хранит контекст и историю взаимодействия.

### Способы создания

- В интерфейсе AI Studio (визуально)
- Через конструктор спецификаций **Yandex Workflows**
- Через **Responses API** или **Realtime API**
- Через опенсорс-фреймворки (OpenAI SDK, LangGraph, LangChain)

### Готовые инструменты

| Инструмент | Описание | Тарификация |
|---|---|---|
| **File Search Tool** | RAG-поиск в загруженных файлах через Vector Store API / Files API | 300 ₽ за 1000 запросов |
| **Web Search Tool** | Поиск в интернете через поисковую базу Яндекса | 915 ₽ за 1000 запросов |
| **MCP Tool** | Подключение к MCP-серверам (внешние API) | Не тарифицируется (только сами MCP-серверы) |
| **Code Interpreter** | Запуск Python-кода в виртуальной среде с сохранением состояния | Не тарифицируется |

### API создания агентов

| API | Назначение | Поддержка |
|---|---|---|
| **Responses API** | Текстовые сценарии. Учитывает контекст переписки, автоматически вызывает инструменты | Все текстовые модели |
| **Realtime API** | Голосовые сценарии. Принимает аудио на вход, синтезирует аудио на выход | Мультимодальные модели + все инструменты |

## MCP Hub

> MCP Hub = функциональность AI Studio для создания, управления и мониторинга **MCP-серверов** (Model Context Protocol).

### Зачем

- Подключать к LLM сторонние инструменты: API, базы знаний, сервисы.
- Один MCP-сервер может содержать **до 50 инструментов**.

### Сценарии

- Подключение к CRM/ERP-системам или внутренним хранилищам
- Автоматизация бизнес-процессов
- Специализированные ассистенты (юридические, медицинские, технические)
- Интеграция с внешними API (новости, курсы валют)

### Транспорт

- **Streamable HTTP** — современный, без необходимости поддерживать постоянное соединение
- **HTTP with SSE** — устаревший, через постоянное HTTP-соединение

### Способы добавления MCP-серверов

1. **Подключение внешнего сервера** (smithery.ai, Yandex Cloud Marketplace, своя VM)
2. **Создание из шаблона** (готовые шаблоны от партнёров)
3. **Создание с нуля**:
   - **Yandex Cloud Functions** — свой код в облаке (auto-scaling)
   - **HTTPS-запрос** — обёртка вокруг существующего API
   - **Yandex Workflows** — рабочий процесс с своей логикой

### Роли

- `serverless.mcpGateways.invoker` — обращаться к MCP-серверам
- `serverless.mcpGateways.anonymousInvoker` — обращаться к публичным MCP-серверам
- `serverless.mcpGateways.editor` — создавать MCP-серверы

### Приватные vs Публичные

- **Приватные**: только агенты с правильной ролью или с IAM-токеном/API-ключом со scope `yc.serverless.mcpGateways.invoke`
- **Публичные**: доступны любым агентам без аутентификации

### Опция логирования

В настройках MCP-сервера можно включить логирование обращений в журнал.

## Альтернатива MCP Hub: Responses API напрямую

> Вместо MCP Hub для обращения к внешнему MCP-серверу из AI-агента можно использовать Responses API. Достаточно указать URL-адрес сервера и API-ключ со scope `yc.serverless.mcpGateways.invoke`.

### Пример (Python)

```python
from openai import OpenAI

client = openai.OpenAI(
    api_key="<API-ключ>",
    base_url="https://ai.api.cloud.yandex.net/v1",
    project="<идентификатор_каталога>"
)

response = client.responses.create(
    model="gpt://<идентификатор_каталога>/yandexgpt",
    input=[
        {"role": "user", "content": "Найди клиента с именем Иван Иванов в CRM"}
    ],
    tools=[
        {
            "server_label": "crm_lookup",
            "server_url": "<URL_MCP-сервера>",
            "type": "mcp",
            "metadata": {"description": "Поиск клиента в CRM по имени"}
        }
    ]
)

print(response.output_text)
```

> ⚠️ **Важно:** в примере используется `gpt://...yandexgpt` модель. Будет ли работать аналогично с Qwen — нужно проверить (Qwen может не поддерживать `tools` через Responses API).

## Применимо ли к Кайросу

**Прямо сейчас (MVP) — нет.** Наша архитектура слоя восприятия делает то же самое, но проще:
- Свой `MessageAnalyzer` вместо tool calling — точно знаем что делаем
- Свой `DossierService` вместо File Search Tool — без векторов, чистый SQL
- Никаких Web Search / Code Interpreter — не нужно для терапевтической поддержки

**Может понадобиться после MVP** для:
- **G8 (агенты PubMed)** — мог бы быть MCP-сервер вместо собственного `PubMedClient`. Но мы уже написали свой, так что не приоритет.
- **G2 (оптимизация)** — File Search Tool может помочь с RAG по `knowledge_base/psychology/grief/*.md`. Но если переходим на свой LoRA (G3) — это всё съест fine-tune.

## Конструктор рабочих процессов (Yandex Workflows)

> Позволяет собирать сложные сценарии на базе AI с помощью готовых шагов автоматизации и управления. Подходит для тех, кто предпочитает визуальные редакторы и Low-code платформы.

Для Кайроса — не наш путь. У нас весь pipeline в Python с тестами. Visual editor усложнит code review и version control.
