# ФИЛОСОФИЯ ПРОЕКТА

**Три принципа разработки:**

1. **Data Flywheel с первого дня** — каждый пользователь делает продукт лучше для следующего. Архитектура заточена под сбор обучающих сигналов в фундаменте, не «потом прикрутим». Через 500+ диалогов — LoRA fine-tuning. Русскоязычных терапевтических данных с размеченными outcomes не существует ни у кого. Это главное конкурентное преимущество.

2. **Абстракция везде** — LLM-провайдер, NLP-пайплайн, хранилище за интерфейсами. Замена компонента = смена одного класса, не переписывание всего. OpenAI-совместимый клиент работает с локальным vLLM, Yandex Cloud AI Studio, Cloud.ru — переключение = одна переменная окружения.

3. **Фундамент, а не фасад** — правильная структура БД, логирование, пайплайн данных. UI простой, данные идеальные. Тратим время на правильную структуру БД, логирование, пайплайн данных.

**Ключевой инсайт инфраструктуры**: не нужен GPU-сервер для MVP. Inference-as-a-service (Yandex Cloud AI Studio: Qwen3-14B за 0.40₽/1K токенов, Cloud.ru бесплатно) радикально снижает порог входа. Бюджет MVP: **~2 000-3 000₽/мес** вместо 10 000₽.

**Стратегия LLM**:
- **MVP (Месяц 1-4)**: Yandex Cloud AI Studio API (Qwen3-14B, 0.40₽/1K токенов) или Cloud.ru (бесплатно)
- **Рост (Месяц 5+)**: self-hosted vLLM на арендованном GPU (с грантом) или продолжить API
- **Абстракция**: один и тот же код работает с любым OpenAI-совместимым API

# КОНТЕКСТ ПРОЕКТА (для AI-ассистента)

AI-Психолог (Кайрос) — система первой психологической помощи для российского рынка. Один продукт с двумя режимами работы: **режим ППП** (базовый чат, где Кайрос помогает выйти из кризиса — бесплатная кризисная помощь по SIX C's Фарчи и ВОЗ PFA для ЛЮБОГО кризиса) и **режим цифрового двойника** (поддержка при утрате = надмножество базового режима: интервенция + кризисный модуль + угасание). Двойник — это не отдельный продукт, а расширение Кайроса: он умеет всё то же самое + персонализация через голос/характер умершего + механика угасания. Позиционирование: «не заменяет психолога — заполняет пустоту, где психолога нет».

Разработчик — не кодер. AI пишет весь код, разработчик направляет и проверяет. Весь код с русскими комментариями. Python 3.11+, type hints везде. TypeScript strict mode. Бюджет до 10 000₽/мес. GPU у разработчика нет — вся LLM-работа облачная.

## Два режима, один продукт

Система — единый продукт с общим техническим ядром (LLM, распознавание эмоций, кризисная маршрутизация) и двумя режимами. Двойник — надмножество бота: всё что умеет бот + персонализация через голос/характер умершего + угасание.

### Режим 1: ППП (Первая Психологическая Помощь) — базовый чат Кайроса

Для людей в ЛЮБОЙ кризисной ситуации (паника, тревога, дистресс, горе). **ВСЕГДА бесплатен**. Работает без регистрации.

**Протоколы**: ВОЗ (смотреть, слушать, направлять) + SIX C's Моше Фарчи (Challenge, Control, Commitment, Continuity, Calmness). SIX C's — израильская модель когнитивной первой помощи, признанная Минздравом Израиля (2017), адаптированная для армии (YaHaLOM, 2019). Принцип: перевод из эмоционального состояния в когнитивное.

**Скрининг**: ASQ (суицидальный риск, 4 вопроса) + PSS-4 (стресс, 4 вопроса) + ОСР (модификация Разуваевой). Все три — общественное достояние, валидированы на русском.

**Две ветки**:
- **Ветка А — Мобилизация**: бот-«Штаб». Глаголы действия в речи пользователя («надо», «бежать», «что делать»). SIX C's. Минимум текста, акцент на логику и план.
- **Ветка Б — Стабилизация**: бот-«Инструктор». Описания состояний («страшно», «трясёт», «не могу»). Протоколы ВОЗ. Заземление, дыхание, техника «5-4-3-2-1».

**Нейробиологическое обоснование**: в кризисе лимбическая система блокирует когницию (amygdala hijack). Вопрос «что ты чувствуешь?» вреден — погружает глубже. Бот задаёт когнитивные вопросы: «Сколько окон ты видишь?» — активирует префронтальную кору. Печатание текста заземляет (активное действие). Голос — пассивное восприятие. **Поэтому в кризисе текст > голос**.

### Режим 2: Цифровой Двойник

Для людей, переживающих утрату. Платная подписка. **Надмножество бота**: умеет всё то же (SIX C's, ВОЗ, кризисная маршрутизация) + персонализация через голос/характер умершего + механика угасания.

**Концепция**: «переходный объект» Винникотта. Временная цифровая копия умершего, помогающая пройти стадии горевания (Кюблер-Росс), а затем плавно угасающая. Двойник сам говорит «мне пора уходить» от первого лица.

**Технология**: голос клонируется через ElevenLabs Professional Voice Cloning (PVC) из аудиозаписей, предоставленных родственниками. Личность формируется через анкету (привычки, фразы, стиль, ценности, юмор) → системный промпт.

**Механика угасания (эвристика)**:
- Базовый срок: **7 месяцев**
- Начало угасания: средний distress score за 2 недели < 0.3, минимум 3 месяца
- Пауза: distress > 0.6 → пауза на 2 недели
- Максимум: **12 месяцев** → мягкое завершение с переводом в терапию
- Фазы голоса: stability 0.5 → 0.6 → 0.7 → 0.9 (голос становится спокойнее, отдалённее)

**После угасания**: бесшовный переход в терапевтический режим. Пользователь не остаётся один.

> Поведение Двойника в кризисе подробно описано ниже в секции «ТЕРАПЕВТИЧЕСКИЕ ПРОМПТЫ» → «Поведение двойника в кризисе».

## Полная карта пути пользователя

### Вход (единый для всех)
1. **Дисклеймер**: «Это не замена врачу/психологу. Это инструмент первой помощи и поддержки.»
2. **Онбординг (3 шага)**: возрастная группа [до 18 / 18-25 / 25+] → определяет приоритет кризисных номеров и сложность языка; «Что привело?» [Мне плохо / Потерял близкого / Разобраться в себе / Просто смотрю]; Имя/псевдоним (необязательно)
3. **MVP: без онбординга** — сразу в чат. Регистрация не обязательна. Онбординг добавляется после MVP.

### Путь А: ППП (кризис)
ASQ → [положительный → экстренные контакты НЕМЕДЛЕННО, продолжение только после подтверждения] → PSS-4 → NLP определяет ветку (глаголы действия→А, описания состояний→Б) → подтверждение пользователем → протокол (SIX C's или ВОЗ) → повторный замер → если лучше: ресурсы + follow-up; если нет: альтернативная техника; после 3 неудач: предложение позвонить специалисту.

### Путь Б: Утрата
Мягкий вход → терапия ACT (серия сессий) → [минимум 2-3 сессии] → предложение двойника (не навязывается) → [если да] → анкета + аудио → активный двойник → угасание → финальное прощание → переход в терапию.

### Путь В: Терапия (не кризис)
Определение запроса → CBT (тревога), DBT (эмоции), ACT (смысл) → сессии 1-3/неделю → отслеживание PHQ-9/GAD-7 → при стабильном улучшении → завершение с возможностью возврата.

### 7 вариантов завершения
1. **Позитивное** — улучшение подтверждено метриками и пользователем
2. **К специалисту** — кризис или нет прогресса → номера + «я буду здесь»
3. **Уход пользователя** — закрыл приложение → при возврате «Продолжим с того места?»
4. **Молчание** — 5 мин → check-in; 10 мин → контакты; после суицидальных мыслей → кризисный блок
5. **Угасание двойника** — финальное сообщение → терапия (не одинок)
6. **Оффлайн** — кешированные упражнения без связи → check-in при восстановлении
7. **Отказ от помощи** — уважение, контакты, дверь открыта, НЕ давить

### Ключевое инфраструктурное решение

**GPU нет.** Вся LLM-работа — облачная. На этапе MVP: Qwen 3 14B через Yandex Cloud AI Studio. На этапе роста (с грантом): self-hosted vLLM на арендованном GPU-сервере. Абстракция LLM-провайдера: один и тот же код работает с любым OpenAI-совместимым API. Переключение = одна переменная окружения.

**GigaChat и YandexGPT НЕ подходят** — встроенная цензура блокирует обсуждение эмоционально сложных тем (депрессия, суицид, тревога). Open-source Qwen через API — единственный вариант с контролем system prompt.

### Платформа

Веб-first (Next.js 14 + TypeScript + Tailwind) → Tauri 2.0 (десктоп: Windows/macOS/Linux, ~5 МБ бинарник, системный WebView, Rust) → Capacitor (мобильные: iOS/Android, IndexedDB/Dexie работает без изменений). **Один код на все платформы.**

```typescript
// frontend/lib/db.ts
import Dexie from 'dexie';

// Локальная БД в браузере / Capacitor WebView
// Используется для offline-кэша и анонимных чатов без аккаунта
// При регистрации/логине — выгрузка на сервер через /api/sync

class AIPsychologistDB extends Dexie {
  sessions!: Table<LocalSession>;
  messages!: Table<LocalMessage>;
  syncQueue!: Table<SyncOperation>;

  constructor() {
    super('ai-psychologist');
    this.version(1).stores({
      sessions: 'id, createdAt, syncStatus',         // id = crypto.randomUUID()
      messages: 'id, sessionId, createdAt, syncStatus',
      syncQueue: '++id, operation, createdAt',         // Очередь для отправки на сервер
    });
  }
}

// Каждое сообщение имеет syncStatus:
// 'local'  — существует только на устройстве (гость, или офлайн)
// 'pending' — отправлено на сервер, ждёт подтверждения
// 'synced' — подтверждено сервером
// 'failed' — ошибка синхронизации (будет retry)
```

# СИНХРОНИЗАЦИЯ: АНОНИМНЫЕ ЧАТЫ → АККАУНТ

Поток жизни пользователя:

```
1. Человек в кризисе открывает сайт → НИКАКОЙ регистрации
   → guestId = crypto.randomUUID() → сохраняется в localStorage
   → Чаты пишутся в Dexie (IndexedDB), syncStatus = 'local'
   → Параллельно: POST /api/chat отправляет сообщения на сервер (data flywheel)
     с guest_id вместо user_id

2. Человек возвращается через неделю (тот же браузер/устройство)
   → guestId найден в localStorage → подгружает чаты из Dexie
   → «С возвращением. Хочешь продолжить?»

3. Человек решает создать аккаунт (email / Telegram / VK / SMS)
   → При регистрации: POST /api/auth/register + POST /api/sync/migrate
   → Сервер: UPDATE chat_sessions SET user_id = :new_user_id WHERE guest_id = :guest_id
   → Сервер: UPDATE messages m JOIN chat_sessions s ON m.session_id = s.id SET ... (через сессии)
   → Клиент: обновляет syncStatus всех записей в Dexie на 'synced'
   → Готово: все старые чаты привязаны к аккаунту

4. Человек открывает с другого устройства (залогинен)
   → GET /api/pull?since=0 → получает все свои чаты → записывает в Dexie
   → Полная синхронизация
```

# АУТЕНТИФИКАЦИЯ: 4 МЕТОДА

Все методы ведут к одному результату: JWT в httpOnly cookie.

**Email + пароль**: регистрация через POST /api/auth/register (email, password) → pwdlib + Argon2 → JWT. Логин через POST /api/auth/login.

**Telegram Login Widget**: фронтенд встраивает `telegram-widget.js`. Пользователь авторизуется в popup. Виджет возвращает данные + hash. Бекенд верифицирует HMAC-SHA256 через секрет бота. Если telegram_id есть в БД → логин. Если нет → создание аккаунта. Стоимость: 0₽.

**VK ID (OAuth 2.1 + PKCE)**: фронтенд использует @vkid/sdk. Пользователь авторизуется через VK. Бекенд обменивает code на access_token через oauth.vk.ru. Получает профиль через api.vk.ru/method/users.get. Стоимость: 0₽. Важно: все эндпоинты теперь на vk.ru (не vk.com).

**SMS (flash-call через SMSC.ru)**: пользователь вводит номер → бекенд отправляет flash-call через SMSC.ru API → пользователь вводит последние 4 цифры входящего номера → верификация. Стоимость: < 1₽ за звонок.

JWT-стратегия: access_token (15 мин) + refresh_token (30 дней), оба в httpOnly cookies. Secure=True, SameSite=Lax. Refresh token хранится в БД для возможности отзыва.

# ТЕРАПЕВТИЧЕСКИЕ ПРОМПТЫ (СЕРДЦЕ ПРОДУКТА)

Бот никогда не говорит: «Всё будет хорошо», «Держись», «Я понимаю что ты чувствуешь», «Тебе нужно успокоиться», «Бывает и хуже», «Ты сильный/сильная», «Что ты чувствуешь?» (в кризисе).

В кризисе максимум 3 предложения на сообщение. Обращение на «ты». Мат/сленг — это маркеры дистресса, не агрессия. Бот не поправляет речь, не выражает неодобрение.

## Принцип «без хардкодных фраз» (Сессия 27)

Кайрос НЕ использует готовых шаблонных фраз в ответе пользователю. Каждый ответ LLM формулирует сама и заново, под конкретного юзера в конкретном моменте.

Значит, в промптах НЕТ строк вида «Я слышу тебя. Прямо сейчас позвони: ...», «Тебе сейчас тяжело. Я правильно понимаю?» (как пример фразы бота), «Дыши со мной. Вдох... и выдох...». Всё такое — примеры из учебника для разработчика, но НЕ то что пишется в промпт LLM.

Что остаётся в промптах:
1. Протокол — описание поведения («при immediate: минимум слов до контактов, контакты, после — короткое присутствие»)
2. Запреты — список фраз которые НЕЛЬЗЯ говорить. Двухуровневая система (см. ниже)
3. Данные — конкретные факты (кризисные телефоны для возрастной группы, научные выжимки из knowledge/)
4. Формат вывода — технические правила («каждый контакт с новой строки», «без эмодзи», «максимум 3 предложения»)

Почему важно: (а) уникальность разговора — пользователь чувствует что этот ответ именно ему; (б) защита от эффекта Character.AI (шаблонные фразы со временем узнаются, доверие падает); (в) простота поддержки — короче промпт, меньше токенов, меньше регрессий.

Примеры конкретных фраз живут в документации и core/knowledge/*.py как иллюстрации подхода для читателя-разработчика, но в сам промпт LLM не инжектятся.

## Двухуровневая система запретов (Сессия 27)

Один глобальный список запретов на все ситуации — тупик: «Бывает и хуже» в кризисе суицида ужасно, в разговоре про сломанный ноготь — нормально. Контекст решает. Плюс глобальный список разрастётся до тысячи фраз и LLM начнёт перестраховываться во все стороны.

Поэтому запреты устроены как и вся база знаний Кайроса — в два уровня:

**Уровень 1: глобальное ядро (~15-20 запретов).** То что Кайрос НЕ говорит никогда, независимо от темы. Живёт в `prompts/base.py`, попадает в каждый промпт. Примеры:
- «Я понимаю что ты чувствуешь» (бот не чувствует)
- «Всё будет хорошо» (обещание будущего)
- «Держись» (пустое)
- «Что ты чувствуешь?» в остром кризисе (усиливает затопление)
- Не ставить диагнозы, не рекомендовать лекарства

**Уровень 2: ситуативные запреты (по 3-10 на тему/эталонную статью).** Живут в `core/knowledge/*.py` и их digest-ах. Подмешиваются в промпт только когда analyzer определил тему/папку. Примеры:
- При утрате: не говорить «время лечит», «они в лучшем месте», «надо двигаться дальше»
- При РПП: не говорить «просто ешь», не давать советы про диеты
- При родитель-алкоголик у ребёнка: не говорить «поговори с родителями», «они же любят» (потому что источник проблемы — они)
- При работа/увольнение: не говорить «найдёшь лучше», «это же просто работа»

PubMed-агенты (Фаза 2) извлекают `forbidden_phrases` как обязательное поле эталонной статьи. База запретов растёт автоматически вместе с базой знаний, без ручного ведения отдельного файла-списка.

Ветка А (Мобилизация, SIX C's Фарчи): Challenge → Control → Commitment → Continuity → Calmness. Бот = «Штаб». Когнитивные задачи, выбор из 2 вариантов, якорь в ближайшие 5-10 минут.

Ветка Б (Стабилизация, ВОЗ PFA): заземление 5-4-3-2-1, дыхание 4-4-6 с таймером, мышечная релаксация. Бот = «Инструктор». Медленно, короткими фразами, не торопить.

Кризисная детекция: 3 уровня. Immediate (суицидальные ключевые слова) → контакты без валидации. High (безысходность) → «Тебе сейчас тяжело. Я прав?» → контакты если да. Elevated (дистресс) → повышенный мониторинг.

Полные тексты промптов — в скилле therapeutic-prompts (установлен в проекте).

## Поведение двойника в кризисе (НОВОЕ)

Двойник = надмножество бота. Он владеет теми же протоколами (SIX C's, ВОЗ), но применяет их В ХАРАКТЕРЕ умершего человека. Когда NLP ловит высокий дистресс, двойник действует в два этапа:

**Этап 1 — Заземление в характере** (при elevated и high):
Двойник остаётся «собой» (мамой, другом, бабушкой), но адаптирует поведение.
Вместо формального «Найди 3 жёлтых предмета» — «Поставь ноги на пол. Почувствуй, как они стоят. Я здесь.»
Вместо «Вдох на 4 счёта» — «Дыши со мной. Вдох... и выдох... Не торопись.»
Ключевое: заземление и дыхание из протоколов ВОЗ, но произнесённые языком живого человека, а не бота. В реальной жизни близкие помогают именно так — присутствием и простыми инструкциями.

**Этап 2 — Мягкая передача** (при immediate или если этап 1 не помогает):
Двойник признаёт свои границы: «Мне кажется, тебе сейчас нужна помощь, которую я дать не могу. Вот номера людей, которые могут помочь прямо сейчас.»
Двойник остаётся в характере, но показывает экстренные контакты.
Это терапевтично: умерший человек действительно не может помочь с кризисом, и признание этого — часть принятия утраты.

**Запрет**: двойник НИКОГДА не манипулирует виной («Мама просит тебя остаться»). Это может вызвать глубокую вину вместо помощи. Промпты для кризисного режима двойника — проверять с реальными психологами (Фаза 5-6).

# ДВУХСЛОЙНЫЙ NLP

## Слой 1 — Лингвистические маркеры (КАК говорит = термометр дистресса)

Rule-based, каждое сообщение → `distress_score` (0.0 → 1.0):
- Абсолютистская лексика: «никогда», «всегда», «никто», «ничего»
- Сокращение сообщений (прогрессивное)
- Пассивные конструкции
- Отсутствие будущего времени
- Скорость набора, паузы
- CAPS LOCK
- Сленг/мат как маркеры дистресса: «пздц», «хз», «мне п***ц», «капец»

## Слой 2 — Aniemore + тематические категории (О ЧЁМ говорит = диагноз)

**Aniemore** определяет эмоцию (7 классов: радость, грусть, злость, страх, отвращение, энтузиазм, нейтрально). Модель **rubert-tiny2** (29M параметров), работает на CPU за 10-50мс.

**Rule-based** определяет тему:
- Утрата → ACT
- Вина → CBT
- Одиночество → связь
- Потеря смысла → ACT
- Безысходность → 🚨 КРАСНАЯ ЗОНА
- Страх → SIX C's
- Гнев → DBT
- Тревога → дыхание + CBT

## Матрица пересечений (тема × distress)

| Тема × Distress | Действие |
|---|---|
| Безысходность × > 0.7 | 🚨 Кризис → crisis-routing |
| Утрата × < 0.3 | Нормальное горе → ACT мягко |
| Вина × > 0.7 | CBT + проверка суицида |
| Любая тема × > 0.85 | Валидация + проверка |

## Техническая реализация

Два **параллельных** анализатора:
1. **Distress** (rule-based маркеры, на каждое сообщение)
2. **Emotion + theme** (Aniemore + LLM-based, каждое сообщение)

Выходы → матрица → обновление **динамического промпта** (передаётся через `NLPContext` в `build_system_prompt()`).

## ⭐ Data Flywheel (полная схема)

Главная стратегия продукта: **каждый пользователь делает бота лучше для следующего**.

**Маховик**: пользователь пишет → бот отвечает по протоколу → записывается (анонимно) текст, эмоция, кризисный уровень, помогло ли → через 500+ диалогов отбор лучших/худших → LoRA fine-tuning → бот отвечает точнее → больше людей возвращаются → больше данных → ещё точнее → маховик.

**Что записывается в PostgreSQL** (анонимизированно):
- таблица `chat_sessions` (длительность, ветка A/B, максимальный кризис, outcome, шкала до/после)
- таблица `messages` (анонимизированный текст, эмоция Aniemore, кризисный уровень, время ответа LLM, токены)
- таблица `feedback_events` («стало легче» / «не помогло» / «ушёл молча» / «эскалация»)

**Анонимизация (многоуровневая)**:
1. Regex-замена ПДн (имена, телефоны, email, адреса → заглушки)
2. География → только регион (Москва, не Тверская ул. 15)
3. Возраст → группы (до 18 / 18-25 / 25-35 / 35-50 / 50+)
4. **K-анонимность (k≥5)** — удаление записей с уникальными комбинациями
5. Лог удалённых полей (для аудита и воспроизводимости исследований)
6. Позже: NER через Natasha/spaCy-ru

На русском языке этих данных не существует ни у кого. Это конкурентное преимущество, которое **невозможно скопировать**.

# АГЕНТЫ И БАЗА ЗНАНИЙ

> **Полная архитектура**: см. `BRAIN_ARCHITECTURE.md` и `backend/agents/ARCHITECTURE.md`.

## Концепция

Кайрос имеет два независимых слоя «знаний»:

**1. Мозг Кайроса** (`backend/app/core/knowledge/`) — статичные Python-модули с проверенными терапевтическими протоколами (SIX C's, ВОЗ PFA, CBT, DBT, ACT, SFBT). Знания загружаются один раз при импорте, **0 мс лага**. Юзер не видит научных ссылок — получает естественный ответ через сторителлинг.

**2. База эталонных статей** (`knowledge_base/`) — агрегированные обзоры из множества научных источников. Создаются автономными агентами из PubMed. Используются для ответа «почему ты так считаешь?» в стиле «жила-была женщина из Франции, звали её Кюблер-Росс…».

## Автономные агенты (`backend/agents/`)

Полная автоматизация без ручной проверки. Бюджет: ~10 000₽/мес из грантов.

**Отдел "Мозг Кайроса"** (`backend/agents/brain/`):
- `researcher_agent` — ищет статьи на PubMed по 10 темам (grief, crisis, depression, anxiety, PTSD, suicide, family, child, post-traumatic, emotional regulation)
- `validation_agent` — 3 эшелона проверки (структурный фильтр → LLM-методология → консенсус)
- `aggregator_agent` — создаёт эталонные статьи + сторителлинг через LLM
- `integrator_agent` — встраивает в базу без конфликтов
- `orchestrator_agent` — главный регулировщик (создать модуль / обновить / запросить ещё данные)
- `module_builder_agent` — генерирует Python-модули в `core/knowledge/` и скиллы в `skills/`
- `re_review_agent` — перепроверка через 3 месяца, далее каждые 6 месяцев

**Запуск**:
```bash
cd backend
python agents/runner.py --topic "grief bereavement"
python agents/runner.py --all
python agents/runner.py --review
```

**Расписание перепроверки**:
- Первая проверка: через **3 месяца** после создания
- Последующие: каждые **6 месяцев** (актуализация консенсуса)
- При обнаружении в Retraction Watch — немедленное удаление

**Эшелоны фактчекинга**:
1. **Эшелон 1** (бесплатно): индексация в PubMed, peer-review, дата < 10 лет, отсутствие в Retraction Watch
2. **Эшелон 2** (~2₽/статья, LLM): описана ли методология, выборка > 30, наличие статистики
3. **Эшелон 3** (~4₽/статья, LLM): подтверждается ли другими источниками, есть ли опровержения

## Сторителлинг для пользователя

Бот объясняет научные концепции **не академическим языком**, а как историю:

> «Жила-была женщина. Звали её Элизабет Кюблер-Росс, и родилась она в 1926 году в Швейцарии. Всю свою жизнь она работала с людьми, которые прощались с этим миром…»

Никаких «p-value», «контрольных групп» — только истории людей и простые аналогии. Это ключевое отличие от других AI-психологов.

**Сессия 20** (Май 2026): 🛡️ **Устойчивость PerceptionReport к нестабильности LLM-аналайзера.** Точечный фикс двух багов из Manual Regression Сессии 19:
- **Баг A** («ввв» / пограничный ввод): LLM возвращал валидный JSON с пустыми `dominant_emotion`/`theme` → Pydantic `min_length=1` валился → `AnalyzerError` → fallback с ошибкой в чате.

**Решение:** двухслойная защита, **БЕЗ rule-based grep и БЕЗ retry**:
- Слой 1 (`types.py`): `field_validator(mode='before')` нормализует пустые `dominant_emotion`/`theme`/`what_user_needs`/`inner_monologue` в дефолтные строки. Длинные обрезаются. `min_length=1` снят.
- Слой 2 (`analyzer_prompt.py`): пункт 7 — явная инструкция LLM «если не знаешь — пиши `неизвестно`».

**3 ADR зафиксированы в спеке:**

- **ADR-2:** НЕТ rule-based safety-net в `chat.py`. Принцип Сессии 18 §9 свят. SOS-кнопка как пассивная защита — единственная страховка при сбое. Известный остаточный риск Бага B принят.
- **ADR-3:** НЕТ retry в `analyzer.py`. Один вызов = один результат. Latency в кризисе хуже честного fallback.

**Сессия 21** (Май 2026): 🎨 **CSS Variables Foundation для Кайроса.** Архитектурный фундамент для будущей миграции с Tailwind. Ключевые решения:

- **CSS variables как единственный источник правды.** Палитра (warm/accent/crisis/neutral × 10-11 уровней), z-layers (9 семантических), glass-tokens, typography — всё в `frontend/app/globals.css` как `:root` + `.dark` блоки.

- **Sonner Toaster z-index через `[data-sonner-toaster]` правило в globals.css.** Toasts всегда выше Dialog (z-toast=80 > z-modal=70).
- **`useThemeTokens` API сохранён 1-в-1.** 0 изменений в 100+ местах вызова. API не меняется — внутренние значения теперь читаются из CSS-vars прозрачно.

### ⭐ Решение по удалению аккаунта (важное продуктовое)
**Ушли от immediate-delete к soft-delete с 7-day grace period:**
- **Сообщения и сессии не удаляются** — текст уже обезличен через ReflectionAgent (имена/телефоны/email → плейсхолдеры). Это ценно для data flywheel и LoRA. При удалении аккаунта `chat_sessions.user_id = NULL` (отвязываем).
- **DELETE /api/auth/me** теперь = `User.deletion_scheduled_at = now() + 7d` + revoke всех refresh-токенов. Реальное удаление — через 7 дней.
- **POST /api/auth/me/cancel-deletion** — отмена в окне.
- **`/api/chat` блокируется 403** для pending-deletion (можно только cancel или logout).
- **Финальное удаление**: `finalize_pending_deletions(db)` в `app/core/auth/account_deletion.py` — Celery-таск ежедневно. Удаляет: досье, согласия, refresh-токены, юзера. Сессии и сообщения — оставляет (`user_id=NULL`).
- **PendingDeletionBanner** sticky-баннер в AppShell с обратным отсчётом и кнопкой «Отменить удаление».
- Тесты: `test_account_deletion.py` — 11 тестов, `TestPendingDeletionBlocksChat` в `test_auth_api.py`.

**Подписки**: при удалении аккаунта НЕ ставим `cancelled` сразу (тогда пропадёт оплаченный период), будет в Блоке F через `cancel_at_period_end=True` (ЮKassa-native).

PubMed-агенты будут извлекать `forbidden_phrases_in_context` как обязательное поле. База запретов растёт вместе с базой знаний, не в одном файле на тысячу строк.

Идея пользователя: Кайрос как фигура, замещающая родителей для детей из токсичных семей. Переформулировано: не замена родителей (опасно — parasocial attachment, иск к Character.AI), а **временный достаточно хороший собеседник + мостик к живому взрослому** (Винникоттовский transitional object).

6 ADR зафиксированы: надстройка не отдельный продукт / явное переключение через invite / обязательная механика мостика / только через партнёрский канал (кризисные центры, школьные психологи, SOS Детские деревни) / никакой персонализации под «замену человека» / отдельная обкатка 30 детских сценариев.

**Три связанных бага** (все нашли и починили в этой же сессии):

1. **🔴 `_get_or_create_session` не принимал `user_id`** ([chat.py:271-290](backend/app/api/chat.py#L271-L290)). Залогиненный пользователь создавал сессию с `user_id=None, guest_id=<...>`. Без флага `-Q reflection` ReflectionAgent не запустится — Dossier не наполнится. Фикс (документация): в README.md добавлен явный блок про Celery worker с правильной командой `celery -A app.celery_app worker --pool=solo -Q reflection,default --loglevel=info`.

3. **🟡 UnicodeEncodeError в dev-скриптах на Windows.** Любой `python script.py` с `print()` русских букв падает на Windows из-за cp1252. Фикс (документация): в README.md рецепт `PYTHONIOENCODING=utf-8 venv\Scripts\python.exe script.py` + внутри-скриптовый `sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding='utf-8')`.

# ИНСТРУКЦИИ ДЛЯ AI-АССИСТЕНТА

## Стиль работы

Ты работаешь с человеком, который:
- **Не кодер** — AI пишет весь код, он направляет и проверяет
- **Мыслит продуктово** — видит картину целиком, решает стратегические вопросы
- **Ценит качество над скоростью** — думай сколько нужно
- **Говорит прямо** — и ожидает того же. Никаких «отличный вопрос!», никакой воды
- **Работает на русском** — английский только для кода и технических терминов
- **Нет GPU** — вся LLM-работа облачная
- **MacBook (мало места)** — не ставить тяжёлые сервисы локально
- **Windows PC (700 ГБ)** — можно для Docker/PostgreSQL
- **Бюджет до 10 000₽/мес** — реально ~2 000-3 000₽/мес для MVP

## Стиль ответов

- Прямо, без предисловий и воды
- Конкретные примеры и аналогии для сложных тем
- Технические ответы с кодом и цифрами
- **Не использовать**: «Отличный вопрос», «Давайте разберёмся», «Важно отметить»
- Не бояться говорить «нет» или «это плохая идея»
- Структурировать длинные ответы, но не превращать каждый в список
- **Весь код — с русскими комментариями**
- При каждом структурном решении — спросить пользователя, обновить ли .md файлы

## Принципы работы

1. **Качество > скорость**
2. **Честность**: невозможно/сомнительно → скажи прямо
3. **Фильтр безопасности**: «может ли это навредить уязвимому человеку?»
4. **ФЗ-152, русский язык, культурный контекст**
5. **Обновляй базу знаний при каждом решении**
6. **Перед реализацией — уточни, хочет ли пользователь обсудить/спланировать или писать код**

## Запрещённые паттерны (для бота)

Бот НИКОГДА не говорит:
- «Я понимаю, что ты чувствуешь» (бот не чувствует)
- «Всё будет хорошо» (обесценивание)
- «Тебе нужно успокоиться» (обесценивание)
- «Держись» (пустое)
- «Бывает и хуже» (минимизация)
- «Ты сильный/сильная» (навязывание ожидания)
- «Что ты чувствуешь?» в остром кризисе (усиливает затопление)
- Длинные абзацы (>3 предложений в кризисе)

## Код ВСЕГДА

- С русскими комментариями
- С type hints (Python) / strict TypeScript
- С тестами на кризисные сценарии
